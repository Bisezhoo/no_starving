## 1. 背景

当前系统进入 Chat 页面时会调用 `GET /api/chat/history` 加载最近 24 小时且最多 60 条完整聊天记录。若后端返回空数组，前端消息列表保持为空。

用户希望当系统进入时没有聊天记录，页面默认展示一段食谱助手介绍语，让用户知道助手能力范围和可提问场景。

## 2. 目标

1. 页面进入后，如果历史消息为空，默认展示一条 assistant 欢迎消息。
2. 页面进入后，如果存在历史消息，只展示真实历史消息，不额外插入欢迎语。
3. 历史接口加载失败时，前端仍展示默认欢迎语，并允许用户继续发送新消息。
4. 默认欢迎语只属于前端 UI 空态，不写入 `chat-history.json`。
5. 默认欢迎语不进入 Agent Memory、LLM 上下文、画像或推荐历史。

## 3. 非目标

1. 不修改后端 `GET /api/chat/history` 响应语义。
2. 不新增后端接口、数据库表、配置项或多语言欢迎语管理。
3. 不把欢迎语持久化为真实聊天记录。
4. 不调整现有消息列表、Markdown 渲染、推荐卡片或 SSE 事件契约。

```markdown
你好！我是你的 食谱助手（Recipe Assistant） 🍳🥗

简单来说，我是一位专门帮你查找各种美味菜肴和饮品食谱的智能助手！无论你想做什么菜、调制什么饮品，我都能帮你找到合适的食谱。

🍽️ 我能做什么？
🥘 搜索菜肴食谱 — 根据菜名、食材、菜系或类别来查找各种美食（中餐、意大利菜、墨西哥菜等）
🍹 查找饮品配方 — 搜索鸡尾酒、无酒精饮料等各种饮品配方
📋 提供详细信息 — 包括食材清单、烹饪步骤、份量等
💡 我可以帮你解决这些场景：
🏠 今晚吃什么？ → 帮你推荐菜肴
🛒 冰箱里有鸡肉和洋葱 → 帮你找能做的菜
🎉 朋友聚会调杯酒 → 帮你查鸡尾酒配方
🥗 想吃素食/低卡/某国菜系 → 帮你精准搜索
有什么想做的菜或想喝的饮品，尽管告诉我吧！😊
```

## 5. 设计方案

采用前端空态欢迎消息方案。

`ChatPage.vue` 保持现有挂载流程：先调用 `loadChatHistory()`，再根据结果初始化 `messages`。新增一个前端常量或小函数生成默认 `AssistantMessage`：

```ts
const DEFAULT_WELCOME_MESSAGE: AssistantMessage = {
  role: "assistant",
  reply: "...欢迎语...",
  cards: [],
  toolCalls: [],
  warnings: [],
  profileUpdates: [],
};
```

为避免后续消息对象被误改影响默认模板，实际赋值时应创建新对象，而不是直接复用同一个引用。

数据流：

```text
ChatPage mounted
  -> loadChatHistory()
  -> history.length > 0：messages = history
  -> history.length === 0：messages = [默认欢迎语]
  -> loadChatHistory() 抛错：messages = [默认欢迎语]

用户发送消息
  -> 在现有消息列表后追加 user 消息和本轮 assistant 占位
  -> 不删除欢迎语
  -> 按现有 SSE 回调更新本轮 assistant 消息
```

## 6. 方案取舍

| 方案 | 优点 | 风险 | 结论 |
|------|------|------|------|
| 前端空态欢迎消息 | 实现最小；不污染真实聊天历史；不改变后端接口语义 | 文案位于前端代码，后续多语言需再抽离 | 采用 |
| 后端历史接口返回默认欢迎语 | 前端更薄 | 会混淆“真实历史”和“空态消息”，后续可能误传给历史恢复逻辑 | 不采用 |
| 首次进入写入 `chat-history.json` | 刷新后一致 | 会把空态文案变成真实聊天记录，违背“没有聊天记录”的语义 | 不采用 |

## 7. 测试方案

| 测试 | 断言 |
|------|------|
| `ChatPage` 历史为空 | 渲染默认欢迎语 |
| `ChatPage` 历史存在 | 渲染历史消息，不渲染默认欢迎语 |
| `ChatPage` 历史接口失败 | 渲染默认欢迎语，发送消息仍可用 |
| 发送消息 | 欢迎语保留，本轮用户消息和 assistant 响应追加到列表 |
