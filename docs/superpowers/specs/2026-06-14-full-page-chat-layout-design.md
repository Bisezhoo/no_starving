# 全页面固定聊天布局设计

| 项目 | 内容 |
|------|------|
| 文档版本 | v0.2 |
| 最近更新时间 | 2026-06-14 22:43:54 CST |

| 关联需求 | 用户要求顶部标题区和底部输入区固定在整个页面上下边缘，中间聊天记录允许滑动 |
| 关联技术设计 | `智能食谱助手技术设计.md` v0.38 |

## 1. 背景

当前 Chat 页面使用三行 Grid：顶部标题区、中间消息列表、底部输入区。由于外层容器使用 `min-height`，页面在内容增多或不同视口高度下可能出现整体滚动，导致用户看到标题区和输入区不稳定贴合页面上下边缘。

用户明确要求固定的是整个浏览器页面的上下边缘，而不是白色聊天面板内部的上下边缘。

## 2. 目标

1. `.topbar` 始终位于页面可视区域顶部。
2. `.composer` 始终位于页面可视区域底部。
3. `.message-list` 占据顶部与底部之间的剩余高度，并独立纵向滚动。
4. 页面整体禁止纵向滚动，避免聊天记录把顶部或底部区域带出视口。
5. 不修改后端接口、SSE 契约、消息数据结构或输入校验规则。

## 3. 非目标

1. 不调整 Chat 消息渲染结构。
2. 不改变欢迎语、流式思考占位、推荐卡片或 Tool 轨迹展示逻辑。
3. 不引入新的布局依赖或第三方滚动库。
4. 不改动后端业务逻辑。

## 4. 方案对比

| 方案 | 优点 | 风险 | 结论 |
|------|------|------|------|
| CSS Grid 三行全屏布局 | 保持现有 DOM 结构；顶部、消息区、输入区仍在同一个布局上下文；不会遮挡消息 | 需要精确设置 `height`、`min-height: 0` 和滚动边界 | 采用 |
| `position: fixed` 固定顶部和底部 | 直观贴边 | 容易遮挡消息列表；需要手工计算上下 padding；移动端键盘场景更脆弱 | 不采用 |
| 让浏览器页面继续整体滚动 | 改动最小 | 不满足顶部和输入区固定在页面上下边缘的要求 | 不采用 |

## 5. 布局设计

`app.css` 作为唯一布局来源：

1. `html`、`body`、`#app` 高度设置为 `100%`，保证 Vue 根节点有稳定视口高度来源。
2. `.app-shell` 使用 `height: 100vh` 和 `overflow: hidden`，页面整体不产生纵向滚动。
3. `.chat-surface` 使用 `height: 100%`，继续采用三行 Grid。
4. `.chat-surface` 的行定义改为 `auto minmax(0, 1fr) auto`，确保中间消息区可以被压缩并产生内部滚动。
5. `.message-list` 设置 `min-height: 0`、`overflow-y: auto` 和 `overscroll-behavior: contain`，把滚动限制在聊天记录区域。
6. 桌面端仅保留左右 24px 外边距；上下方向不保留外边距，让标题区和输入区贴住整个页面的上边缘和下边缘；移动端继续占满全屏。

## 6. 测试设计

1. 新增前端 CSS 布局契约测试，读取 `frontend/src/styles/app.css`，验证：
   - 根节点高度链路存在。
   - `.app-shell` 使用固定视口高度并隐藏页面滚动。
   - `.chat-surface` 使用 `auto minmax(0, 1fr) auto` 三行布局。
   - `.message-list` 使用 `min-height: 0` 和纵向滚动。
2. 运行现有 `chat-page.spec.ts`，验证消息发送、历史加载和占位逻辑不受布局变更影响。
3. 运行 `npm run build`，验证 CSS 与 TypeScript 构建通过。
4. 使用浏览器打开本地页面，检查顶部和底部区域在桌面视口内固定，中间消息区独立滚动。

## 7. 风险与处理

| 风险 | 处理 |
|------|------|
| `min-height: 0` 缺失导致 Grid 子项撑开页面 | CSS 契约测试固定该规则 |
| 移动端高度使用 `100vh` 受浏览器地址栏影响 | 维持首版简单实现；如后续发现移动端浏览器地址栏跳动，再单独评估 `100dvh` 兼容策略 |
| 外层 padding 与“贴页面边缘”的理解冲突 | 本次按用户最新确认处理：上下方向贴整个页面边缘；桌面左右仍保留当前视觉约束 |
