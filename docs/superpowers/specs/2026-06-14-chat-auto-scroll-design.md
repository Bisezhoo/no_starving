# 聊天消息自动滚动到底部设计

| 项目 | 内容 |
|------|------|
| 文档版本 | v0.2 |
| 最近更新时间 | 2026-06-14 22:55:53 CST |
| 文档状态 | 已确认 |
| 关联需求 | 用户要求每次有新的消息就滚动到最底部 |
| 关联技术设计 | `智能食谱助手技术设计.md` v0.40 |

## 1. 背景

Chat 页面已经采用全页面三段式布局：顶部标题区固定在页面顶部，底部输入区固定在页面底部，中间 `.message-list` 独立滚动。当前发送新消息、加载历史或流式回复变长时，消息列表不会主动滚动到底部，用户可能需要手动下拉才能看到最新内容。

## 2. 目标

1. 页面加载历史消息或默认欢迎语后，消息列表滚动到底部。
2. 用户发送新消息并插入 assistant 占位消息后，消息列表滚动到底部。
3. assistant 流式回复、错误、卡片、Tool 轨迹或画像更新导致消息内容变化时，消息列表继续滚动到底部。
4. 自动滚动逻辑集中在 `MessageList.vue`，`ChatPage.vue` 不增加滚动控制职责。

## 3. 非目标

1. 不增加“用户手动上滑时暂停自动滚动”的复杂策略。
2. 不引入虚拟列表、滚动库或动画库。
3. 不修改后端接口、SSE 契约或消息数据结构。

## 4. 方案对比

| 方案 | 优点 | 风险 | 结论 |
|------|------|------|------|
| `MessageList.vue` 内部监听 `messages` 并滚动到底部 | 滚动是列表展示职责；调用点集中；覆盖历史加载、发送、流式更新 | 深度监听会随消息内容变化触发，但当前最多 60 条，成本可接受 | 采用 |
| `ChatPage.vue` 在每个 SSE 回调后调用 `scrollToBottom()` | 触发时机显式 | 滚动逻辑分散在 `delta/card/done/error/catch` 多处，后续维护成本高 | 不采用 |
| 仅在发送时滚动 | 实现最小 | 流式内容、卡片展开或历史加载后仍可能不在底部 | 不采用 |

## 5. 设计

`MessageList.vue` 增加列表 DOM 引用：

1. 模板中 `.message-list` 增加 `ref="listEl"`。
2. 组件中使用 `watch(() => props.messages, scrollToBottom, { deep: true, flush: "post" })`。
3. `onMounted()` 时调用一次 `scrollToBottom()`，覆盖初始历史或欢迎语渲染。
4. `scrollToBottom()` 只在 `listEl.value` 存在时执行，把 `scrollTop` 设置为 `scrollHeight`。

该方案不影响消息内容、消息顺序或输入状态，只改变滚动位置。

## 6. 测试设计

1. 新增 `frontend/tests/message-list.spec.ts`。
2. 挂载 `MessageList`，为 `.message-list` 设置可控的 `scrollHeight`，验证挂载后 `scrollTop` 等于底部。
3. 更新响应式 `messages` 数组，调整 `scrollHeight`，验证新增消息后再次滚到底部。
4. 修改 assistant 消息的 `reply`，调整 `scrollHeight`，验证流式内容变化后继续滚到底部。

## 7. 风险与处理

| 风险 | 处理 |
|------|------|
| 深度监听频繁触发 | 当前历史上限 60 条，首版可接受；后续如果引入大量消息或虚拟列表，再设计“接近底部才自动滚动”策略 |
| jsdom 不计算真实 `scrollHeight` | 单测通过 `Object.defineProperty()` 控制滚动高度，验证组件写入 `scrollTop` 的行为 |
| 用户手动查看旧消息时被拉回底部 | 本次用户明确要求每次新消息滚动到底部；暂停自动滚动属于后续增强，不混入本次 |
