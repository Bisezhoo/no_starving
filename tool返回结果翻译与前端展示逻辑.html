Tool 返回结果翻译与前端展示逻辑

本文梳理当前代码中的“Tool 返回结果如何被解释/本地化”以及“前端如何接收 SSE 并渲染回复和推荐结果”。当前实现不是把外部 API 的所有字段逐字翻译，而是在保留原始字段的基础上，增加 localized* 辅助展示字段，并要求 LLM 最终文本回复使用本轮用户语言。

1. 结论摘要

问题 当前逻辑 结果
用户语言怎么确定？ 后端用 detect_locale(message) 按本轮消息判断；包含中文字符则为 zh-CN，否则为 en-US。 不保存 languagePreference，每轮单独判断。
Tool 返回结果是否直接翻译？ Tool 层不翻译；Tool 只返回标准化 MealCard / CocktailCard。 原始标题、食材、步骤、分类等字段保留外部 API 原文。
卡片怎么本地化？ Agent 在推荐排序后调用 localize_cards()，写入 localizedLanguage、localizedSummary，必要时写入 localizedInstructions。 前端展示中文/英文辅助解释，但不覆盖原始字段。
最终自然语言回复怎么保证语言？ OpenRouter prompt 中注入 Output locale；Tool 结果回填后再次要求最终回复使用本轮语言。 最终 delta / done.reply 依赖 LLM 按语言约束生成。
前端是否做翻译？ 不做。前端只渲染后端给的 reply、cards、localized* 字段。 业务翻译/解释逻辑留在后端。

2. 后端：Tool 返回结果的本地化链路

1. 语言检测

Agent 开始时执行 detect_locale(message)，得到 detectedLocale。

2. LLM 调 Tool

OpenRouter 根据 prompt 生成 tool_calls，Tool 参数必须是英文。

3. Tool 返回标准卡片

ToolRunner 执行 Tool，返回 ToolDataResult.cards。

4. Agent 本地化并发 SSE

排序后调用 localize_cards()，通过 card 和 done.cards 返回。

2.1 语言检测

位置 规则 影响范围
backend/app/domain/language.py 本轮消息中存在中文字符，则 zh-CN；否则 en-US。 meta.detectedLocale、LLM prompt、本地化卡片字段。

2.2 LLM 回复语言约束

在 OpenRouterStepLlm._build_messages() 中，后端会向 system prompt 注入：

  • Output locale: zh-CN/en-US
  • 要求自然语言回复、解释、warning、追问都使用该语言。

当已经有 Tool 结果时，后端还会追加一条 user message，明确要求：

  • Final reply must use {detectedLocale}
  • Tool 结果、空结果、校验失败和下一步说明都用本轮语言解释。

这里约束的是 LLM 的自然语言输出。外部 API 原始字段不会被 LLM 改写，也不会在 Tool 层被替换。

2.3 卡片本地化字段

字段 生成位置 当前含义 是否覆盖原始字段
localizedLanguage card_localization.py 记录本轮用于展示解释的语言，如 zh-CN。 否
localizedSummary card_localization.py 根据卡片类型生成一句本轮语言的解释摘要。中文时是“推荐 XXX：这是一道/一款...”格式。 否
localizedInstructions card_localization.py 仅当用户本轮问“怎么做 / 步骤 / 详细 / how to / cook / make”等时生成。中文时给每步加 步骤 N： 前缀。 否
localizedTitle 模型字段预留 当前代码没有生成实际值，通常为 null 或未展示。 否

2.4 当前“翻译”的边界

会本地化的内容

  • LLM 文本回复：delta / done.reply
  • 卡片辅助摘要：localizedSummary
  • 必要时的步骤辅助展示：localizedInstructions
  • warning / 追问：由 prompt 要求 LLM 使用本轮语言

不会强制翻译的内容

  • 外部 API 原始标题：title
  • 原始食材名和计量：ingredients[].name/measure
  • 原始步骤正文：instructions
  • 分类、国家、酒精属性、杯型等外部枚举字段

3. 后端：SSE 事件中的结果返回

事件 来源 数据含义 前端用途
meta Agent 请求开始 requestId、detectedLocale 当前前端没有直接展示，但可用于调试或后续显示。
delta OpenRouter 流式文本 LLM 生成的增量文本，语言由 prompt 约束。 追加到助手消息 reply。
tool_call Agent 调用 Tool 前 Tool 名称、参数、状态。 进入 Tool 轨迹列表。
tool_result ToolRunner 执行后 Tool 状态、耗时、结果数量、错误、校验错误，可能包含原始标准化卡片。 进入 Tool 轨迹列表；不是主要卡片展示来源。
card Agent 排序和本地化后 当前可展示的 cards 快照，已包含 localized* 辅助字段。 更新助手消息 cards，触发结果区展示。
profile_update 用户偏好解析 画像 patch 和原因。 进入画像变化区。
error 运行期失败 错误码和错误文案。 设置助手消息 error。
done Agent 收尾 最终 reply、cards、toolCalls、warnings、detectedLocale。 用最终快照覆盖消息状态，清除 pending。

4. 前端：接收结果和渲染逻辑

1. 发起请求

ChatPage.send() 插入用户消息和 pending 助手消息。

2. 解析 SSE

streamChat() 用 ReadableStream 解析事件。

3. 更新状态

不同事件回调更新 reply、cards、toolCalls、warnings。

4. 展示消息

AssistantMessage 渲染文本；有卡片时渲染结果区。

4.1 ChatPage 的事件处理

SSE 事件 前端处理 展示结果
delta message.reply += data.text，并清除 pending。 文本回复实时增长。
card message.cards = data.cards，并清除 pending。 显示推荐结果区。
tool_call 追加一条 Tool started 轨迹。 Tool 面板可看到调用开始。
tool_result 追加 Tool 结果轨迹。 Tool 面板可看到状态、耗时、结果数或错误。
profile_update 追加到 profileUpdates。 画像变化区展示 patch。
error 设置 message.error，清除 pending。 展示错误文案。
done 用最终快照设置 reply/cards/toolCalls/warnings，清除 pending。 最终状态定稿。

4.2 AssistantMessage 的展示顺序

  1. 如果有 warnings，先展示 warning 条。
  2. 如果 pending=true 且没有 reply/error/cards，展示“思考中”。
  3. 如果有 reply，用 MarkdownReply 渲染并通过 DOMPurify 清洗。
  4. 如果有 error，展示错误文案。
  5. 只有 cards.length > 0 时才展示 AssistantResultBlock。

4.3 推荐结果区的字段优先级

区域 组件 字段优先级
卡片标题 MealCard / CocktailCard 使用 card.title；当前不使用 localizedTitle。
卡片摘要 MealCard / CocktailCard 如果存在 localizedSummary，展示它。
分类/国家/酒精/杯型 卡片 meta row 展示原始外部字段：category、country、alcoholic、glass。
食材 卡片和步骤区 展示原始 ingredients[].name 和 measure。
步骤 AssistantResultBlock.stepsFor() 优先 localizedInstructions；没有则回退 instructions。

4.4 推荐结果收起/展开

  • 默认最多展示前 3 张卡片。
  • 超过 3 张时显示“查看更多”。
  • 展开后以横向滑动轨道展示全部卡片，并显示“左右滑动查看更多”。
  • 食材/步骤区使用同一个 visibleCards，未展开的卡片步骤不会提前露出。

5. 历史消息加载逻辑

阶段 逻辑 注意点
页面挂载 ChatPage.onMounted() 调用 loadChatHistory()。 如果历史为空，显示默认欢迎消息。
后端读取 GET /api/chat/history 返回 chat-history.json 中最近 24 小时、最多 60 条消息。 返回的是已保存的最终 UI 快照，不重新翻译。
避免覆盖 如果用户在历史返回前已经发送消息，则不再用历史覆盖当前 messages。 避免竞态导致新消息消失。

6. 关键边界与后续可优化点

当前系统的“翻译”更准确地说是“后端本地化辅助解释”：保留外部 API 原始事实字段，同时增加本轮语言的摘要和步骤前缀。它没有对所有标题、食材、步骤做完整机器翻译。
  • 如果产品要求“所有卡片字段完整中文化”，需要新增后端翻译层或让 LLM 对卡片生成完整 localized copy，并标记机器翻译来源。
  • 如果要支持多语言，需要把 card_localization.py 的模板从 zh-CN/en-US 扩展为更多 locale。
  • 如果要翻译步骤正文，不能只加 步骤 N： 前缀；需要逐步翻译并保留原文。
  • 前端当前不做翻译，这是符合“业务逻辑归后端”的边界。
