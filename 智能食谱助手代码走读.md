## 1. 实现范围

本次实现基于 `Python FastAPI + Vite Vue 3`，覆盖首版单用户、单 Agent、单活跃请求的智能食谱助手：

1. 后端提供 `POST /api/chat/stream` SSE 流式接口和 `/api/health` 健康检查。
2. 后端集成 OpenRouter DeepSeek 模型、TheMealDB、TheCocktailDB、ToolRunner、轻量用户画像、推荐历史、Agent Memory 和完整聊天历史。
3. 前端实现 Chat 输入、全页面固定三段式布局、消息列表自动滚动、空聊天默认欢迎语、Agent 文本回复、流式思考占位、流式开始顶部提示、按需结构化结果区、推荐结果默认收起/横向滑块、Tool 调用默认折叠、菜谱/饮品卡片概要展示和单卡详情弹窗。
4. 本地 JSON 文件作为首版画像、推荐历史、Agent Memory 和最近一天完整聊天记录的持久化来源，不引入数据库。
5. 后端补齐结构化日志与敏感字段脱敏，完整用户消息、完整 `sourceText`、完整系统提示词和敏感 Header 默认不记录。
6. 后端新增 Tool 结果统一翻译链路：卡片进入 SSE 前通过 `CardTranslationService` 生成并校验 `localized*` 补丁，失败时回退原始卡片。

## 2. 后端走读

### 2.1 应用入口与 API

`backend/app/main.py`：

- `create_app()` 负责创建 FastAPI 应用、注册路由、创建 `ConversationLock`。
- 注入 `agent` 时用于测试；未注入时通过 `build_default_agent()` 装配真实 Agent。
- OpenRouter 配置缺失时不使用占位成功响应，而是记录 `startup_error`，健康检查返回 503，Chat 返回配置错误事件。

`backend/app/api/chat.py`：

- Controller 只做请求字段校验、单活跃请求锁获取、SSE 响应返回。
- 空消息返回 HTTP 400，超长消息返回 HTTP 400，并发请求返回 HTTP 409。
- SSE 建立后的运行期异常通过 `error` 事件返回，并在 `finally` 中释放 `conversation_lock`。
- 新增 `GET /api/chat/history`，读取 `chat-history.json` 中最近 24 小时且最多 60 条完整消息。
- `POST /api/chat/stream` 在收到 `done` 或 `error` 后保存本轮用户消息和助手最终快照，不改变原有 SSE 事件契约。

`backend/app/api/health.py`：

- 正常返回 `{ code: 200, message: "success", data: { status: "ok" } }`。
- 启动配置不可用时返回 HTTP 503，不泄露密钥或完整异常。

### 2.2 配置、SSE 与错误

`backend/app/core/config.py`：

- 读取 OpenRouter、外部 API 地址、Agent 步数预算和敏感日志开关。
- `OPENROUTER_API_KEY`、`OPENROUTER_MODEL` 必填，空值会校验失败。

`backend/app/core/logging.py`：

- 提供 `log_event()` 统一输出 JSON 结构化日志。
- 默认将 `userMessage`、`sourceText`、`systemPrompt` 记录为 `{ present, length }` 摘要。
- `LOG_FULL_USER_MESSAGE`、`LOG_FULL_SOURCE_TEXT`、`LOG_FULL_SYSTEM_PROMPT` 分别只放开对应字段。
- `Authorization`、`apiKey`、`token`、`secret` 等敏感字段始终脱敏，即使开启 `LOG_SENSITIVE_EXTERNAL_HEADERS` 也不输出完整密钥。
- `summarize_tool_arguments()` 对含中文的 Tool 参数做摘要，避免校验失败时把用户原文派生内容写入日志。

`backend/app/core/sse.py`：

- `format_sse(event, data)` 统一序列化事件，使用紧凑 JSON，保留中文。

`backend/app/core/errors.py`：

- 提供 `BusinessException` 与统一 JSON 错误响应结构。

### 2.3 领域层

`backend/app/domain/models.py`：

- 定义 `MealCard`、`CocktailCard`、`TasteProfile`、`RecommendationRecord`、`AgentMemory`、`ToolDataResult`、`SseEvent` 等核心模型。
- 新增 `UserChatHistoryMessage`、`AssistantChatHistoryMessage` 和 `ChatHistoryMessage`，用于完整 UI 历史恢复。
- `MealCard` / `CocktailCard` 新增 `localizedIngredients`、`localizedCategory`、`localizedCountry`、`localizedAlcoholic`、`localizedGlass` 等展示辅助字段；原始 `title`、`ingredients`、`instructions` 仍完整保留。

`backend/app/domain/language.py`：

- 按本轮用户消息是否包含中文字符检测 `zh-CN` / `en-US`，不持久化语言偏好。

`backend/app/domain/card_localization.py`：

- 在 Tool 返回卡片进入 `card` 事件和 `done.cards` 前，按本轮 `detectedLocale` 生成 `localizedLanguage` 与 `localizedSummary`。
- 用户本轮询问“怎么做”“步骤”“详细”或英文 how-to 意图时，为详情卡片生成 `localizedInstructions`，但不覆盖外部 API 原始 `instructions`。

`backend/app/domain/tool_args.py`：

- 负责 Tool 参数英文约束、allowlist 校验、`idMeal` 来源约束和无效 JSON 拦截。
- `ingredient` / `query` 含中文时返回可重试校验失败，避免下游 TheMealDB / TheCocktailDB 接收非英文参数。

`backend/app/domain/taste_profile.py`：

- 从自然语言中提取轻量画像信号，例如喜欢/不吃食材、清淡偏好、禁酒偏好。

`backend/app/domain/recommendation.py`：

- 对候选卡片做硬过滤、同 ID 去重、主食材/分类/菜系软降权。
- 推荐历史最多保留 20 条，用于降低连续多日重复推荐同类食材的概率。

### 2.4 外部 API、Tool 和 Agent

`backend/app/adapters/*_client.py`：

- 封装 TheMealDB / TheCocktailDB HTTP 请求。
- 支持 `OUTBOUND_HTTP_PROXY` 和 `OUTBOUND_HTTP_TRUST_ENV`，本地或服务器需要出站代理时由后端统一传入 httpx。
- `limit` 不传递给外部 API，只在后端候选截取和补详情阶段生效。

`backend/app/tools/meal_tool.py`、`backend/app/tools/cocktail_tool.py`：

- `search_meals` 支持 query、ingredient、category、area 和随机推荐。
- `get_meal_detail` 只按可信来源的 `idMeal` 查询详情。
- `search_cocktails` 支持禁酒硬过滤，用户明确不喝酒时过滤 Alcoholic 饮品。

`backend/app/services/tool_runner.py`：

- 统一执行 Tool：先校验参数，再调用 Tool，最后包装状态、耗时、结果数、校验错误或异常。
- 入参校验失败时不调用外部 API，返回 `validation_failed` 给 Agent。
- 记录 `tool_run_finished`，包含 `requestId`、Tool 名称、状态、耗时、结果数、补详情数量和脱敏后的参数摘要。

`backend/app/services/openrouter_client.py`：

- `OpenRouterClient` 封装 OpenRouter Chat Completions streaming 请求。
- 与 Tool 外部客户端共享出站代理配置，避免本地调试时 OpenRouter 请求绕过代理导致 SSE 长时间只有 `meta` 事件。
- 记录 `openrouter_request_started`、`openrouter_request_finished` 和 `openrouter_request_failed`，Header 经过统一脱敏。
- `OpenRouterStepLlm.stream_step()` 边读取 OpenRouter 流边产出 `delta`，最终汇总 `tool_calls`。
- Tool 参数 JSON 解析失败时标记为 `__invalid_json__`，交给 Tool 校验层拦截，不转成伪造参数。
- `_build_messages()` 会把 `detectedLocale` 动态写入 system prompt；Tool 结果回填消息也显式要求最终回复使用本轮输出语言，避免 Tool 后回复退回英文。

`backend/app/services/card_translation_service.py`：

- `CardTranslationService` 是 Tool 结果统一翻译与校验入口，接收已经排序去重后的 `Card[]`、`detectedLocale` 和详情意图标记。
- 先调用现有 `card_localization.py` 生成模板兜底字段，再在目标语言不是英文且存在 translator 时调用 LLM 翻译。
- `OpenRouterCardTranslator` 复用 OpenRouter streaming client，请求模型返回 JSON patch 数组；不让模型返回完整卡片。
- `CardTranslationPatch` 使用 Pydantic `extra="forbid"` 和 `StrictStr` 校验，只允许 `localized*` 字段。
- 校验规则包括：ID 必须来自输入卡片、禁止未知字段、`localizedIngredients` 长度必须与原始 `ingredients` 一致、`localizedInstructions` 只能在详情意图下出现且长度一致、所有本地化字符串必须非空且不超长。
- 单张卡片补丁失败只丢弃该卡片补丁，其他合法补丁继续合并；翻译服务异常时返回模板本地化结果和非阻塞 warning。
- 英文输出场景跳过 LLM 翻译，避免不必要的模型调用。

`backend/app/services/default_agent.py`：

- 装配默认 Agent：OpenRouter、ToolRunner、Meal/Cocktail Client、MemoryStore 和 Tool schema。
- 从 `Settings` 读取出站代理配置，并传给 OpenRouter、TheMealDB、TheCocktailDB 三类外部 HTTP 客户端。
- 使用同一个 `OpenRouterClient` 同时装配 `OpenRouterStepLlm` 和 `OpenRouterCardTranslator`，保证默认 Agent 具备卡片翻译能力。
- 前端不接触 OpenRouter Key；所有外部依赖都在后端创建。

`backend/app/services/agent_orchestrator.py`：

- 负责 ReAct 主循环：LLM 推理、Tool 调用、重复参数缓存、步数预算、卡片事件、最终 `done` 快照。
- 每轮开始加载画像、推荐历史和 Agent Memory；根据用户本轮消息更新画像并发送 `profile_update`。
- Tool 返回卡片后进行推荐排序，再调用 `CardTranslationService` 做本地化 patch 校验与合并，发送 `card` 事件，并在 `done.cards` 返回最终快照。
- 翻译服务返回的 warning 会合并到本轮 `done.warnings`，不阻断卡片展示、画像更新或推荐历史写入。
- 传给 LLM 的 context 包含 `detectedLocale`，确保初次推理和 Tool 结果回填后的最终回复都能收到明确语言约束。
- 每轮结束写回画像、推荐历史和 Agent Memory；memory 不保存完整用户原文，只保存轻量摘要和候选卡片引用。
- 记录 `agent_request_started` 和 `agent_request_finished`，包含请求 ID、消息长度、检测语言、画像/历史概况、Tool 数量、卡片数量、耗时和 warning 数量。

### 2.5 并发与持久化

`backend/app/services/conversation_lock.py`：

- 保护单用户单 Agent 的单活跃请求。
- 已有请求处理中时，Chat API 返回 HTTP 409，不进入 Agent。

`backend/app/services/memory_store.py`：

- 使用本地 JSON 保存 `taste-profile.json`、`recommendation-history.json`、`agent-memory.json`。
- 采用临时文件 + `Path.replace()` 原子替换。
- 不再额外增加 JSON 写锁；并发写由外层 `conversation_lock` 保证。
- JSON 损坏读取失败记录 `memory_store_read_failed`，写入失败记录 `memory_store_write_failed`，不记录原始文件内容。

`backend/app/services/chat_history_store.py`：

- 使用本地 `chat-history.json` 保存完整 UI 聊天消息。
- 读取时跳过损坏 JSON、非法消息和超过 24 小时的消息。
- 写入时追加本轮消息后同时执行最近 24 小时和最多 60 条消息裁剪。
- 写入失败返回 warning，不抛出到 Chat 主流程。
- JSON 损坏读取失败记录 `chat_history_read_failed`，写入失败记录 `chat_history_write_failed`，不记录原始聊天正文。

## 3. 前端走读

`frontend/src/api/sse.ts`：

- 解析 SSE 文本流，支持多事件、多行 data 和尾部残留 flush。

`frontend/src/api/chat.ts`：

- 使用 `fetch + ReadableStream` 调用 `POST /api/chat/stream`。
- 先处理非 2xx HTTP JSON 错误，再解析 SSE。

`frontend/src/components/ChatPage.vue`：

- 页面状态容器，维护用户消息、助手消息、发送状态、本轮 assistant `pending` 状态和 SSE 事件回调。
- 页面挂载时加载最近 24 小时、最多 60 条历史消息；历史为空或加载失败时展示默认食谱助手欢迎语，不阻断发送。

## 4. 核心链路

1. 前端提交消息到 `POST /api/chat/stream`。
2. 后端校验 `message`，获取 `conversation_lock`。
3. Agent 加载本地 JSON 中的画像、推荐历史和 memory，检测本轮语言，合并本轮画像信号，并把 `detectedLocale` 写入 LLM context。
4. Agent 发送 `meta`，前端收到后在当前助手消息顶部展示“疯狂思考中，请稍等”，必要时后端继续发送 `profile_update`。
5. OpenRouter 以流式方式返回文本 delta 或 tool call，Prompt 中包含本轮输出语言约束。
6. Agent 对 Tool 参数做英文归一化与校验，ToolRunner 执行 TheMealDB / TheCocktailDB Tool。
7. Tool 返回卡片后，后端做推荐排序、重复推荐降权、禁酒硬过滤，再通过 `CardTranslationService` 合并已校验的 `localized*` 补丁，发送 `card`。
8. Agent 在预算内继续推理，Tool 结果回填消息再次声明输出语言，最终发送 `done`，包含最终回复、卡片、Tool 轨迹和 warnings。
9. 后端更新推荐历史和 Agent Memory，原子写回 JSON。
10. Chat API 根据 `done` 或 `error` 事件保存本轮完整聊天消息到 `chat-history.json`，再释放 `conversation_lock`。

页面进入历史加载链路：

1. `ChatPage.vue` 挂载时调用 `loadChatHistory()`。
2. 后端 `GET /api/chat/history` 读取 `chat-history.json`。
3. `ChatHistoryStore` 过滤损坏消息、超过 24 小时的消息，并只返回最近 60 条。
4. 前端用返回消息初始化 `messages`，继续复用现有 `MessageList`、`AssistantMessage` 和卡片组件渲染。
5. 如果返回空数组或历史接口失败，前端用本地 `AssistantMessage` 展示默认欢迎语；该消息不持久化。

## 6. 审查关注点

1. `AgentOrchestrator` 的职责已经覆盖 ReAct、卡片返回和持久化，后续如果继续扩展复杂意图，建议再拆分专门的 memory/recommendation 协调服务。
2. OpenRouter tool calling 的流式解析已支持参数分片合并，但还需要在真实模型下做端到端联调，确认 DeepSeek 模型返回结构与测试样本一致。
3. `MemoryStore` 基于单进程 + `conversation_lock` 设计，多进程部署前必须改为数据库或共享锁。
4. `ChatHistoryStore` 同样是单用户、本地 JSON 方案；多用户、多 session 或多实例部署前需要升级为数据库或带用户维度的共享存储。
5. 当前没有实现账号、多用户、多 session，也没有非流式降级接口，符合首版边界。
6. npm 安装时提示依赖漏洞，需要后续单独评估升级策略，避免为了本次首版实现扩大变更范围。
7. 本次新增 `marked` 和 `DOMPurify` 后，`npm install` 仍提示 6 个审计漏洞；未执行 `npm audit fix --force`，避免破坏性依赖升级混入本次展示修复。
8. Tool 结果卡片的 `localized*` 字段来自后端翻译补丁或模板兜底，不覆盖外部 API 原始 `title`、`instructions` 和 `ingredients`；审查时需重点看 patch 校验是否足够严格，避免 LLM 改写原始事实。
9. 日志默认不输出完整用户消息、完整 `sourceText`、完整系统提示词或完整敏感 Header；排障打开敏感日志开关时仍需避免长期留存日志。
