## 1. 摘要

本次任务是基于免费公开菜谱与饮品 API，构建一个智能食谱助手。用户通过 Chat 对话描述想吃什么，后端 Agent 调用工具搜索菜谱、获取详情、推荐饮品，并把结果以结构化卡片返回给前端展示。

接口测试数据表明，TheMealDB 与 TheCocktailDB 的详情字段存在大量 `null`、空字符串、空格和重复随机结果。因此，本需求的核心不是简单转发外部 JSON，而是由后端完成意图识别、工具调用、数据清洗、去重、错误降级和结构化输出。

## 2. 关键假设

1. 当前项目技术栈已确认为后端 Python FastAPI、前端 Vite + Vue 3 + TypeScript；本需求分析只定义业务边界、接口契约、数据处理规则和测试目标，不进入代码实现。
2. OpenRouter API Key 通过后端环境变量配置，首版通过 OpenRouter 使用 DeepSeek 系列模型，具体模型 ID 由后端启动参数配置，前端不可接触模型配置和密钥。
3. 全接口测试已覆盖 TheMealDB / TheCocktailDB 的 `search`、`filter`、`lookup`、`random` 和 TheMealDB `categories`；需求以 `data_interface_testing/full_api/api-full-test-report.md` 的跨接口结论为准。
4. 首版优先完成必做项：3 个 Agent Tool、多轮对话、结构化展示。加分项按阶段进入。
5. 首版目标用户可以使用中文或英文提问；后端必须识别提问语言，把查询意图转换为外部 API 可接受的英文参数，并按提问语言解释或翻译输出内容。
6. 首版为单用户、单 Agent、单活跃请求，不需要支持多用户、多 `session`、账号体系、并发会话合并、跨设备同步或会话隔离。
7. 菜品推荐基于当前唯一用户的轻量口味画像，不做复杂推荐系统、账号级长期画像或跨用户学习；首版使用后端本地 JSON 文件持久化画像、推荐历史和 Agent Memory，不引入数据库。

## 4. 背景

任务要求使用 TheMealDB 和 TheCocktailDB 两个公开免费 API，结合 OpenRouter 大模型能力，构建一个能理解自然语言的食谱推荐助手。用户不需要知道具体 API 参数，只要表达“想吃鸡肉”“推荐一顿晚餐”“想要素食”之类的需求，Agent 应能选择合适工具并返回可读、可点击、可展示的结果。

为什么需要数据分析先行：接口测试材料显示外部 API 字段并不稳定。食材字段是 `strIngredient1~20` 或 `strIngredient1~15` 的编号字段，后段大量为空；用量字段常有前后空格；标签、视频、来源、多语言说明大量缺失；随机结果会重复。因此系统必须先建立统一的数据模型，前端只消费清洗后的结构。

## 5. 目标与成功标准

### 5.1 业务目标

1. 用户能通过自然语言完成菜谱搜索、菜谱详情查看、饮品搜索和餐食搭配推荐。
2. Agent 能自主决定是否调用 `search_meals`、`get_meal_detail`、`search_cocktails`。
3. 前端能展示结构化结果，包括图片、名称、分类、国家/地区、食材/配料、做法摘要和来源信息；普通 Agent 回复不应被结果卡片或页签容器包裹。
4. 系统面对外部 API 空值、重复、超时、无结果时能优雅降级。
5. 系统能识别用户本轮提问语言，并用相同语言生成主要回复、结果解释和必要翻译；外部 API 原始英文名称保留为辅助信息。
6. 系统能从对话和显式配置中维护轻量口味画像，并基于画像解释推荐原因。

## 6. 用户与场景

| 用户场景 | 示例输入 | 期望行为 |
|----------|----------|----------|
| 按关键词找菜 | `我想吃 chicken` | Agent 搜索菜谱，返回若干菜谱卡片 |
| 按食材找菜 | `家里有鸡蛋和番茄，能做什么` | Agent 优先按主要食材搜索，必要时追问 |
| 查看菜谱详情 | `第一个怎么做` | Agent 使用上一轮上下文中的菜谱 ID 获取详情 |
| 推荐一顿饭 | `给我推荐一顿晚餐` | Agent 推荐主菜，并搭配一个饮品 |
| 饮食偏好 | `想吃素一点的` | Agent 优先按 Vegetarian/Vegan 或相关关键词过滤 |
| 饮品推荐 | `推荐一杯清爽的鸡尾酒` | Agent 搜索饮品，展示配料、杯型和制作说明 |
| 中文关键词查询 | `我想吃鸡肉，不要太辣` | Agent 提取偏好，将核心食材转换为英文查询词，如 `chicken` |
| 会话偏好延续 | `我想吃素` 后再问 `再推荐一个` | 当前唯一会话内继续沿用素食偏好 |
| 英文提问 | `I want a light chicken dinner` | Agent 使用英文回复，并解释推荐理由 |
| 中英混合提问 | `想吃 chicken，配什么 drink` | Agent 识别主要语言为中文，用中文解释，Tool 参数保持英文 |
| 画像提取 | `我不吃香菜，喜欢清淡一点` | Agent 更新轻量口味画像，后续推荐避开香菜并偏清淡 |
| 画像解释 | `再推荐一道适合我的` | Agent 基于当前画像推荐，并说明匹配了哪些偏好 |
| 避免重复推荐 | `这周别总给我推荐牛肉` | Agent 查询推荐历史，降低近期牛肉菜的推荐优先级 |

## 7. 价值主张

1. 用户不用学习 API 查询方式，用自然语言就能得到食谱建议。
2. 推荐结果不是原始 JSON，而是清洗后的结构化内容，可直接阅读和操作。
3. 系统能把菜谱和饮品组合起来，降低用户“吃什么、配什么”的选择成本。
4. 后端统一处理 API 不稳定字段，前端展示更稳定，后续扩展也更简单。

### 8.2 必需 Tool

| Tool | 输入 | 行为 | 输出 |
|------|------|------|------|
| `search_meals` | `query`、`ingredient`、`category`、`area`、`limit` | 调用 TheMealDB 搜索或筛选接口；`filter` 返回 5 字段摘要，后端只对返回数组中最多 `limit` 条候选调用 `lookup` 补详情 | 标准化菜谱摘要或详情列表 |
| `get_meal_detail` | `idMeal` | 调用 TheMealDB `lookup.php?i={id}` | 标准化菜谱详情 |
| `search_cocktails` | `query`、`ingredient`、`limit` | 调用 TheCocktailDB 搜索或筛选接口；`filter` 返回 3 字段摘要，后端只对返回数组中最多 `limit` 条候选调用 `lookup` 补详情 | 标准化饮品摘要或详情列表 |

说明：TheMealDB 的 `filter` 接口返回 `idMeal`、`strMeal`、`strMealThumb`、`strArea`、`strCountry` 5 个字段；TheCocktailDB 的 `filter` 接口只返回 `idDrink`、`strDrink`、`strDrinkThumb` 3 个字段。两个外部 API 均没有 `total`、`limit`、`page` 或游标能力，后端无法验证外部返回是否为全部匹配结果。`limit` 是本系统 Tool 的返回与补详情上限，不传递给外部 API；如果外部 API 返回更多结果，后端只基于单次外部调用结果做去重、排序、最多 `limit` 条截取和必要的 `lookup` 补详情，避免响应过慢和请求放大。

无结果处理必须兼容三种格式：TheMealDB 所有接口返回 `{"meals": null}`，TheCocktailDB `search`/`lookup` 返回 `{"drinks": null}`，TheCocktailDB `filter` 顶层返回 `{"drinks": "no data found"}`。后端统一转为标准空数组和清晰回复文案。

Tool 调用前必须做语言处理：Agent 先检测提问语言，再提取意图、食材、菜系、饮品偏好，并转换为英文 API 参数。`ingredient` 和 `query` 下游只能接收英文，不能把中文原词透传给 TheMealDB / TheCocktailDB；例如用户输入“鸡肉晚餐”时，`search_meals` 的 `ingredient` 使用 `chicken`，回复和卡片摘要使用中文。

英文参数必须保留来源信息，便于校验和调试：`ingredient` / `query` 可以来自用户原文翻译、当前 Agent Memory 或候选卡片；`category` / `area` 必须来自外部 API 支持的英文枚举或明确映射；`idMeal` / `idDrink` 只能来自上一轮候选卡片或 Tool 结果，不能由 LLM 自行生成。转换结果、来源片段和检测到的输出语言应进入 Tool 调用轨迹，但前端默认只展示参数摘要，不展示完整用户原文。

Tool 入参校验失败时，后端应把结构化错误返回给 Agent，让 LLM 基于错误信息最多修正 1 次参数。修正只能使用用户原始问题、当前 Agent Memory、候选卡片、字段枚举和校验错误；如果仍无法生成可信英文参数，应停止 Tool 调用并追问用户，不允许为了凑结果编造食材、菜名、地区、分类或 ID。

首版 Tool 不做多条件组合查询。`search_meals` 同时存在多个查询参数时，后端只选择一个主查询条件，优先级为 `ingredient > category > area > query > random`；`search_cocktails` 的优先级为 `ingredient > query > random`。未被选为主查询的条件只作为后端排序、过滤或推荐理由信号，不触发额外外部接口合并。

### 8.4 轻量口味画像

首版使用当前唯一会话内的轻量口味画像，服务于菜品推荐和多轮偏好延续。画像由后端维护，来源包括用户自然语言表达、可选表单输入和 Agent 对上一轮偏好的结构化提取。

| 字段 | 类型 | 来源 | 用途 |
|------|------|------|------|
| `dietaryRestrictions` | string[] | 用户表达或表单 | 过滤素食、纯素等饮食限制 |
| `likedIngredients` | string[] | 用户表达 | 推荐时优先匹配喜欢的食材 |
| `dislikedIngredients` | string[] | 用户表达 | 推荐时避开不喜欢或不吃的食材 |
| `preferredCuisines` | string[] | 用户表达或地区偏好 | 映射 TheMealDB `area` / `country` 查询 |
| `flavorPreferences` | string[] | 用户表达 | 记录清淡、 spicy、甜、酸等口味倾向 |
| `allowAlcohol` | boolean | 用户表达或默认策略 | 决定是否推荐含酒精饮品 |

`lastIntent` 不属于画像字段，归入 Agent Memory，用于支持“再推荐一个”“换个清淡的”等追问。`recommendationHistory` 不属于画像字段，独立持久化到 `recommendation-history.json`，用于短期防重复和主食材/分类/菜系轮换。

画像与推荐历史规则：

1. 显式表达优先于推断，例如“我不吃牛肉”立即写入 `dislikedIngredients`。
2. 新消息可覆盖旧偏好，例如“今天可以喝酒”覆盖本轮饮品推荐的 `allowAlcohol`。
3. 未设置 `allowAlcohol` 时，含酒精饮品默认可进入候选，但回复不得主动鼓励饮酒；用户表达“不喝酒”“无酒精”“不要酒精”等意图时，立即将 `allowAlcohol=false` 作为硬过滤条件。
4. 画像只在当前唯一会话内有效，不做账号级长期保存。
5. 推荐结果必须能解释至少一个匹配信号，例如“避开了香菜”“优先选择了素食分类”。
6. 画像只做偏好层面的辅助推荐，不做过敏、医疗或营养精确判断。
7. 推荐历史采用纵向数组对象，不使用并列数组，避免菜品、食材、分类和时间错位。
8. 推荐历史默认只做短期去重和多样性轮换，依赖后端本地 `recommendation-history.json` 持久化；后端数据未清空时，跨天仍可避免连续推荐同一主食材。
9. 推荐策略区分硬边界和软边界：`dislikedIngredients`、明确饮食限制、`allowAlcohol=false` 属于硬过滤，不能为了凑结果而突破；近期重复 `itemId` 属于推荐场景硬去重；近期主食材、分类、菜系高频属于软降权，候选不足时可以回退返回最高分结果，并说明已经尽量换口味。

### 8.5 完整聊天记录

完整聊天记录用于恢复前端 UI 历史，不作为 LLM 上下文的真实来源。Agent 多轮推理仍依赖轻量 `AgentMemory`、候选卡片引用、推荐历史和画像。

记录规则：

1. 后端使用独立 `chat-history.json` 保存完整聊天消息，不写入 `agent-memory.json`。
2. 保留窗口为最近 24 小时；超过 24 小时的消息不得在页面进入时返回。
3. 数量上限为最近 60 条消息，用户消息和助手消息各算 1 条。
4. 每轮正常完成后保存本轮用户消息和助手最终消息快照，包括 `reply`、`cards`、`toolCalls`、`warnings`、`profileUpdates` 和可选 `error`。
5. 运行期错误如果已经形成助手错误消息，可保存用户消息和助手错误消息；客户端中途断开时不保证保存未完成的部分消息。
6. 历史接口只做读取和裁剪，不触发 LLM、Tool 调用、画像合并、推荐历史追加或 Agent Memory 更新。
7. 当历史接口返回空数组时，前端可展示默认欢迎语作为 UI 空态；该欢迎语不是后端返回的真实历史消息。

## 9. 数据接口分析结论

### 9.0 全接口测试总览

`data_interface_testing/full_api/api-full-test-report.md` 对 11 个接口做了 30 组参数采样，结论如下：

| 接口类型 | TheMealDB | TheCocktailDB | 需求影响 |
|----------|-----------|---------------|----------|
| `search` | 返回 54 字段详情，30 词搜索得到 332 条去重结果 | 返回 51 字段详情，30 词搜索得到 162 条去重结果 | 可直接标准化为详情卡片 |
| `filter` | 返回 5 字段摘要，包含 `idMeal`、名称、图片、`strArea`、`strCountry` | 返回 3 字段摘要，只包含 `idDrink`、名称、图片 | 结构化展示前只对最多 `limit` 条候选补 `lookup` |
| `lookup` | 返回 54 字段详情 | 返回 51 字段详情 | 详情页和 `filter` 补全的权威来源 |
| `random` | 返回 54 字段详情，可能重复 | 返回 51 字段详情，可能重复 | 推荐时需按 ID 去重 |
| `categories` | 返回 14 个分类，4 字段 | 不适用 | 可用于分类过滤候选项 |

全接口测试还确认：两个 API 的 `filter` 参数不区分大小写，但后端仍应统一转换、URL 编码并记录实际调用参数。

### 9.1 TheMealDB

| 数据点 | 需求影响 |
|--------|----------|
| 详情共 54 个字段，12 个从未缺失 | 只把核心字段作为稳定展示字段 |
| `search`、`lookup`、`random` 返回 54 字段详情 | 可直接清洗为详情卡片 |
| `filter.php?i/c/a` 返回 5 字段摘要 | 列表可先展示基础信息；需要食材/步骤时必须补 `lookup` |
| `categories.php` 返回 14 个分类，字段完整 | 可作为分类筛选的候选来源 |
| `strMealAlternate`、`strImageSource`、`strCreativeCommonsConfirmed` 永远缺失 | 不进入首版 UI |
| `strIngredient1~3` 从未缺失，后续食材缺失率逐步升高 | 遍历 `1~20`，跳过空值，不固定展示数量 |
| `strMeasure` 常带前后空格 | 后端统一 `trim()` |
| `strInstructions` 99/100 含换行，格式不统一 | 后端保留原文并拆成可渲染段落，前端不要自行解析步骤编号 |
| `strTags` 65% 缺失 | 标签仅作为可选展示 |
| 全接口测试中 `strCountry` 零缺失，`strArea` 在 `search/filter/random` 中可能缺失 | 首选 `strCountry`，缺失时回退 `strArea` |
| 无结果返回 `{"meals": null}` | 后端统一转为空数组 |
| `search.php?s=` 空参数会返回 25 条默认数据 | 只有明确做“默认推荐”时才可使用，普通搜索不得误传空字符串 |
| 随机接口 100 次出现 5 个重复 ID | 结果列表按 `idMeal` 去重 |

### 9.2 TheCocktailDB

| 数据点 | 需求影响 |
|--------|----------|
| 详情共 51 个字段，11 个从未缺失 | 只依赖 `idDrink`、`strDrink`、`strDrinkThumb`、`strCategory`、`strAlcoholic`、`strGlass`、`strInstructions` 等核心字段 |
| `search`、`lookup`、`random` 返回 51 字段详情 | 可直接清洗为详情卡片 |
| `filter.php?i` 只返回 3 字段摘要 | 搜索饮品配料时只对最多 `limit` 条候选调 `lookup`，用于展示分类、酒精类型、杯型、配料和步骤 |
| `strIngredient10~15`、`strMeasure10~15` 在采样中永远缺失 | 仍遍历 `1~15`，但 UI 不预留固定 15 行 |
| `strInstructionsZH-HANS`、`strInstructionsZH-HANT` 在全接口测试中 99%~100% 缺失 | 中文输出不得依赖 API 中文字段，必须由 Agent 基于英文说明解释或翻译 |

| `strAlcoholic`、`strGlass`、`strCreativeCommonsConfirmed` 在详情接口中零缺失 | 饮品卡片可稳定展示酒精类型和杯型；CC 字段可作为可选来源信息 |
| `strMeasure` 有大量前后空格 | 后端统一 `trim()` |
| `strGlass` 存在大小写不一致 | 展示原值，筛选或聚合时再做归一化 |
| `search/lookup` 无结果返回 `{"drinks": null}`，`filter` 无结果时顶层返回 `{"drinks": "no data found"}` | 后端统一判空，不能只判断数组或 null |
| `search.php?s=` 空参数返回 null | 饮品默认推荐不得使用空搜索，应使用 `random.php` 或明确关键词 |
| 随机接口 100 次出现 5 个重复 ID | 结果列表按 `idDrink` 去重 |

## 10. API 契约建议

### 10.1 Chat 接口

`POST /api/chat/stream`

首版唯一对外 Chat 接口为流式接口，响应类型为 `text/event-stream`。暂不实现 `POST /api/chat/message` 非流式接口；如果后续部署环境证明 SSE 被网关或代理缓冲，再评估新增非流式接口。

请求体：

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| `message` | string | 是 | `trim` 后长度 1~1000 |

画像和 Agent Memory 不由前端随请求传入。后端从内存状态或本地 JSON 文件读取 `TasteProfile`、推荐历史和 Agent Memory，并在用户通过自然语言表达偏好或产生新候选结果后更新本地 JSON。

前置校验失败或当前已有请求处理中时，不建立 SSE 流，直接返回统一 HTTP JSON 错误：空消息、超长消息返回 `400`；`conversation_lock` 已被占用时返回 `409` 和“当前已有请求处理中，请稍后再试”。SSE `error` 事件只用于流已经建立后的运行期异常或部分能力失败。

流式响应事件至少包含 `meta`、`delta`、`tool_call`、`tool_result`、`card`、`profile_update`、`done`、`error`。`card` 事件和 `done.cards` 统一使用 `{ cards: Card[] }` 结构，数组元素通过 `type: "meal" | "cocktail"` 区分卡片类型；后端负责输出统一卡片契约，前端只按 `type` 分发渲染。前端展示时，`delta` / `done.reply` 始终作为文本消息输出，只有收到 `card` 事件或 `done.cards.length > 0` 时，才在该 Agent 消息内追加结构化结果区块。`done` 事件携带最终完整快照，包括 `detectedLocale`、`reply`、`tasteProfile`、`profileUpdates`、`cards`、`toolCalls` 和 `suggestions`。

SSE 请求建立失败时，前端展示连接失败；SSE 中途断开且未收到 `done` 时，前端保留已收到内容并提示“生成中断，请重试”。首版不自动改走非流式接口，避免重复 Tool 调用、重复画像更新和幂等复杂度。

### 10.2 后端校验规则

| 对象 | 校验规则 |
|------|----------|
| 用户消息 | 非空，最大 1000 字符，去除首尾空格 |
| Tool `limit` | 整数，范围 1~10，默认 5；仅限制后端返回候选和补详情数量，不传递给 TheMealDB / TheCocktailDB |
| 菜谱 ID / 饮品 ID | 非空数字字符串，最大 20 位 |
| 搜索关键词 | 非空时最大 100 字符，调用外部 API 前 URL 编码；普通搜索不得把空字符串透传给外部 API |
| 后端本地画像 JSON | 数组长度、单项长度、布尔类型和推荐历史条目全部由后端校验；去重、去空格、忽略空字符串；文件不存在时使用空画像 |
| 外部 API 响应 | 兼容数组、`null`、顶层 `drinks: "no data found"`；统一标准化为空数组或卡片列表 |

### 10.3 语言策略

| 项目 | 首版策略 |
|------|----------|
| 用户输入 | 支持中文、英文和中英混合；后端检测本轮主要提问语言 |
| 外部 API 查询 | 优先使用英文关键词、英文食材、英文分类和地区参数 |
| 输出语言 | 不接收 `languagePreference` 或 `locale` 入参；按本轮主要提问语言输出 |
| Agent 回复 | 使用输出语言解释推荐理由、筛选依据、注意事项和下一步建议 |
| 卡片标题 | `title` 永远保留外部 API 英文原名；`localizedTitle` 仅作为按输出语言生成的辅助展示，不替代原名 |
| 卡片摘要 | 列表推荐默认生成 `localizedSummary`，按输出语言解释推荐理由、核心食材和适配偏好 |
| 配料/食材翻译 | `ingredients` 永远保留标准化后的原始食材/用量；`localizedIngredients` 只保存同结构本地化副本，数组长度和字段类型必须与原始食材列表一致 |
| 做法说明 | `instructions` 永远保留外部 API 原始英文步骤；`localizedInstructions` 只在详情场景生成，例如用户问“怎么做”“第一个详细步骤” |
| 本地化语言字段 | `localizedLanguage` 等于本轮 `detectedLocale`，只描述本轮输出，不进入用户画像，不作为长期语言偏好持久化 |
| Tool 原始结果 | Tool 只负责查询、清洗和标准化，不直接翻译返回值 |
| 统一翻译 | 后端在推荐排序、去重和批次前置完成后，对最终准备展示的卡片执行统一本地化；一轮对话可按批次、长度或新增候选多次调用 LLM 翻译，不限制只能调用一次 |
| 同语言跳过 | 当待翻译源内容语言与本轮输出语言一致时，不调用翻译 LLM，直接使用原始字段或已有本地化字段 |
| 翻译校验与回退 | LLM 只能返回本地化补丁，后端校验 ID、字段白名单、数组长度、字段类型和长度限制；校验失败的补丁丢弃，卡片仍按原始内容展示 |
| 前端展示 | 前端按 `localizedTitle ?? title`、`localizedIngredients ?? ingredients`、`localizedInstructions ?? instructions` 等优先级展示，不承担翻译逻辑 |
| 中英混合 | 以主要提问语言为输出语言；若难以判断，优先沿用当前唯一会话最近一次语言 |
| 风险提示 | 机器翻译或摘要可能不等同原始菜谱，应避免用于过敏、医疗、营养精确判断 |

### 10.4 Tool 结果统一翻译与校验

Tool 返回的标准化结果可能仍以英文为主。首版不把翻译逻辑放进 `search_meals`、`get_meal_detail` 或 `search_cocktails`，而是在后端聚合层统一处理：

1. 处理顺序为 `Tool 标准化结果 -> 推荐排序/去重/新批次前置 -> 本地化翻译补丁 -> SSE card/done.cards`。
2. 翻译目标只包含展示字段，例如 `localizedTitle`、`localizedSummary`、`localizedIngredients`、`localizedInstructions`、`localizedCategory`、`localizedCountry`、`localizedAlcoholic`、`localizedGlass`；不得修改 `id`、`type`、`title`、`ingredients`、`instructions` 等原始字段。
3. LLM 翻译返回值必须是结构化补丁数组，后端按卡片 ID 合并；不能让 LLM 返回完整卡片，避免覆盖原始事实。
4. 翻译补丁必须通过后端模型校验：字段必须在白名单内，数组长度必须与原始数组一致，字符串长度受限，类型必须保持不变。
5. 单轮可按新增卡片批次或 payload 长度多次调用翻译 LLM，但每次调用都必须独立校验和可降级；翻译失败不得阻断主回复、卡片展示、画像更新或推荐历史记录。
6. 如果源语言和目标语言一致，或卡片缺少可翻译内容，则跳过翻译调用。

### 10.5 单用户、单会话与画像作用域

| 项目 | 首版策略 |
|------|----------|
| 用户身份 | 单用户匿名使用，不做登录注册 |
| 会话数量 | 单 Agent 会话，不支持多 `session` 并行 |
| 会话标识 | 不需要对外暴露 `sessionId` |
| 并发边界 | 单活跃请求；如果两个页面同时发送消息，第二个请求应返回“当前已有请求处理中，请稍后再试” |
| 多轮上下文 | 当前唯一会话内保存最近消息、候选卡片、轻量口味画像和 Agent Memory |
| 画像作用域 | 当前会话级轻量口味画像，使用 `TasteProfile` 字段 |
| 画像与 Memory 生命周期 | 技术设计默认由后端本地 JSON 文件保存单用户画像、推荐历史和 Agent Memory；后端每轮读取、校验、合并并写回 |
| 非目标 | 首版不做多用户隔离、账号级长期画像、跨设备同步、家庭成员多画像 |

## 11. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | Chat 首包在合理时间返回；补详情时最多处理 `limit` 条候选，避免一次查询放大过多外部请求 |
| 可靠性 | 外部 API 设置超时；失败时返回可理解文案；可对详情结果按 ID 做短期缓存 |
| 安全 | OpenRouter Key 仅在后端环境变量；完整 API Key 始终不记录，系统提示词、完整用户消息、完整 `sourceText` 和外部敏感 Header 默认不记录，可通过启动参数在调试场景显式开启 |
| 幂等 | 查询类请求不产生业务状态；当前会话上下文更新只影响唯一 Agent 会话 |
| 并发控制 | 首版只允许单活跃请求；`conversation_lock` 必须在正常完成、异常、超时和客户端断开时释放，避免死锁 |
| 可观测性 | 核心链路记录请求 ID、Tool 名称、外部 API、耗时、结果数量、异常类型 |
| 可维护性 | 所有外部 API 字段清洗集中在后端适配层，前端不读取 `strIngredient1` 这类原始字段 |
| 隐私 | 日志默认不记录完整对话、完整用户输入、完整 `sourceText` 或外部敏感 Header；如需排障，可通过启动参数显式开启对应敏感日志，完整 API Key 仍必须脱敏 |
| 完整聊天历史 | 后端只保留最近 24 小时、最多 60 条 UI 消息；该历史用于页面恢复，不作为 LLM 上下文输入 |
| 外部请求放大控制 | `filter` 补 `lookup` 详情时只处理外部单次返回数组中的最多 `limit` 条候选，并记录补详情数量和失败数量 |

## 13. 非目标

1. 首版不做用户登录、收藏、购物清单、营养计算和支付。
2. 首版不承诺外部 API 存在完整本地化数据。解释、摘要和翻译由 Agent 基于原始内容生成。
3. 首版不做复杂饮食禁忌医学建议，只做用户偏好层面的过滤和提醒。
4. 首版不要求所有加分项同时完成。流式输出和 Tool 链路可视化可后置。
5. 首版不做账号体系、多用户、多 `session`、用户级长期偏好、跨设备同步和多成员画像。
6. 首版不承诺所有菜名、步骤、配料都有人工级专业翻译。
