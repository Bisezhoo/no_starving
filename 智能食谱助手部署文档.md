## 1. 部署目标

本项目首版采用 `FastAPI 后端 + Vite Vue 3 前端静态资源 + Nginx 同源反向代理` 的部署方式。

部署完成后应满足：

1. 用户通过浏览器访问前端页面。
2. 前端以同源路径调用 `POST /api/chat/stream`。
3. Nginx 将 `/api/*` 转发到 FastAPI 后端。
4. 后端读取 OpenRouter Key 和模型配置，调用 OpenRouter、TheMealDB、TheCocktailDB。
5. 后端在本地 JSON 文件中持久化轻量用户画像、推荐历史和 Agent Memory。

## 2. 部署架构

```text
Browser
  |
  | HTTPS
  v
Nginx
  |-- /                 -> frontend/dist 静态文件
  |-- /api/*            -> FastAPI uvicorn:127.0.0.1:8000

FastAPI
  |-- OpenRouter        -> DeepSeek 模型
  |-- TheMealDB         -> 菜谱数据
  |-- TheCocktailDB     -> 饮品数据
  |-- backend/data/*.json
```

首版不引入数据库、Redis、多用户账号、多 session 或非流式降级接口。

## 3. 运行环境要求

| 组件 | 建议版本 | 说明 |
|------|----------|------|
| Python | 3.11+ | 后端运行 FastAPI、httpx、pydantic |
| Node.js | 20 LTS+ | 前端构建 Vite Vue 应用 |
| npm | 10+ | 安装前端依赖 |
| Nginx | 1.20+ | 托管前端静态文件，反代 SSE API |
| 网络 | 可访问外网 | 后端需访问 OpenRouter、TheMealDB、TheCocktailDB |

## 4. 目录约定

以下示例假设代码部署在：

```bash
/opt/no_starving
```

关键目录：

```text
/opt/no_starving/backend
/opt/no_starving/frontend
/opt/no_starving/frontend/dist
/opt/no_starving/backend/data
```

`backend/data` 保存本地持久化 JSON：

```text
taste-profile.json
recommendation-history.json
agent-memory.json
```

该目录需要被后端运行用户写入，并且应纳入备份范围。

## 5. 后端配置

后端配置通过环境变量读取。生产建议使用 systemd `EnvironmentFile` 或进程管理平台的环境变量管理，不要把真实密钥提交到 Git。

### 5.1 必填配置

| 变量 | 示例 | 说明 |
|------|------|------|
| `OPENROUTER_API_KEY` | `sk-or-v1-***` | OpenRouter API Key |
| `OPENROUTER_MODEL` | `deepseek/deepseek-chat` | OpenRouter 上的 DeepSeek 系列模型 |

### 5.2 可选配置

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `OPENROUTER_BASE_URL` | `https://openrouter.ai/api/v1` | OpenRouter API 地址 |
| `MEALDB_BASE_URL` | `https://www.themealdb.com/api/json/v1/1` | TheMealDB API 地址 |
| `COCKTAILDB_BASE_URL` | `https://www.thecocktaildb.com/api/json/v1/1` | TheCocktailDB API 地址 |
| `OUTBOUND_HTTP_PROXY` | 空 | 后端访问 OpenRouter、TheMealDB、TheCocktailDB 时使用的显式出站代理 |
| `OUTBOUND_HTTP_TRUST_ENV` | `true` | 是否允许 httpx 读取进程环境中的代理配置 |
| `AGENT_MAX_TOOL_CALLS` | `6` | 单轮 Chat 最大 Tool 调用数 |
| `AGENT_MAX_LLM_STEPS` | `4` | 单轮 Chat 最大 LLM 推理步数 |
| `LOG_FULL_SYSTEM_PROMPT` | `false` | 是否记录完整系统提示词 |
| `LOG_FULL_USER_MESSAGE` | `false` | 是否记录完整用户消息正文 |
| `LOG_FULL_SOURCE_TEXT` | `false` | 是否记录完整 Tool sourceText |
| `LOG_SENSITIVE_EXTERNAL_HEADERS` | `false` | 是否记录外部请求敏感 Header |
| `LOG_STACK_TRACE` | `false` | 是否记录完整异常堆栈 |

敏感日志开关默认必须保持关闭。即使在排障时开启，也不得把包含密钥或完整用户隐私内容的日志长期保留。

### 5.3 日志与排障

后端使用 JSON 结构化日志，常见事件包括：

| 事件 | 说明 |
|------|------|
| `app_startup` | 应用启动配置摘要和日志开关状态 |
| `agent_request_started` / `agent_request_finished` | 单轮 Agent 请求开始和结束 |
| `tool_run_finished` | Tool 执行完成，包含状态、耗时、结果数和参数摘要 |
| `openrouter_request_started` / `openrouter_request_finished` / `openrouter_request_failed` | OpenRouter 请求链路 |
| `memory_store_read_failed` / `memory_store_write_failed` | 画像、推荐历史或 Agent Memory JSON 读写异常 |
| `chat_history_read_failed` / `chat_history_write_failed` | 最近聊天记录 JSON 读写异常 |

默认日志策略：

1. `userMessage`、`sourceText`、`systemPrompt` 默认只记录 `{ present, length }` 摘要。
2. `Authorization`、`apiKey`、`token`、`secret` 等字段始终脱敏。
3. `LOG_FULL_USER_MESSAGE`、`LOG_FULL_SOURCE_TEXT`、`LOG_FULL_SYSTEM_PROMPT` 只放开对应字段，不会联动放开其他敏感内容。
4. `LOG_STACK_TRACE=true` 只建议在受控排障窗口临时开启。

### 5.4 示例环境文件

示例路径：

```bash
/etc/no-starving/backend.env
```

内容示例：

```bash
OPENROUTER_API_KEY=sk-or-v1-replace-me
OPENROUTER_MODEL=deepseek/deepseek-chat
OUTBOUND_HTTP_PROXY=
OUTBOUND_HTTP_TRUST_ENV=true
AGENT_MAX_TOOL_CALLS=6
AGENT_MAX_LLM_STEPS=4
LOG_FULL_SYSTEM_PROMPT=false
LOG_FULL_USER_MESSAGE=false
LOG_FULL_SOURCE_TEXT=false
LOG_SENSITIVE_EXTERNAL_HEADERS=false
LOG_STACK_TRACE=false
```

## 6. 后端部署步骤

进入后端目录：

```bash
cd /opt/no_starving/backend
```

创建虚拟环境：

```bash
python3 -m venv .venv
```

安装依赖：

```bash
.venv/bin/pip install -e .
```

初始化持久化目录：

```bash
mkdir -p /opt/no_starving/backend/data
```

启动后端：

```bash
set -a
. /etc/no-starving/backend.env
set +a
.venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000
```

本地健康检查：

```bash
curl -i http://127.0.0.1:8000/api/health
```

期望返回：

```json
{"code":200,"message":"success","data":{"status":"ok"}}
```

如果 OpenRouter 配置缺失或为空，健康检查会返回 HTTP 503。

## 7. systemd 服务示例

示例文件：

```bash
/etc/systemd/system/no-starving-backend.service
```

内容示例：

```ini
[Unit]
Description=No Starving FastAPI Backend
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/opt/no_starving/backend
EnvironmentFile=/etc/no-starving/backend.env
ExecStart=/opt/no_starving/backend/.venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000
Restart=on-failure
RestartSec=5
User=no-starving
Group=no-starving

[Install]
WantedBy=multi-user.target
```

启动服务：

```bash
sudo systemctl daemon-reload
sudo systemctl enable no-starving-backend
sudo systemctl start no-starving-backend
sudo systemctl status no-starving-backend
```

查看日志：

```bash
sudo journalctl -u no-starving-backend -f
```

## 8. 前端部署步骤

进入前端目录：

```bash
cd /opt/no_starving/frontend
```

安装依赖：

```bash
npm ci
```

构建生产静态文件：

```bash
npm run build
```

构建产物位于：

```text
/opt/no_starving/frontend/dist
```

当前前端默认请求同源接口 `/api/chat/stream`，因此生产部署推荐用 Nginx 同域名承载前端并反代 `/api/*`。如果前端和后端拆成不同域名，当前代码需要额外设计 CORS 和 API Base URL 配置后再部署。

## 9. Nginx 配置示例

示例文件：

```bash
/etc/nginx/sites-available/no-starving.conf
```

内容示例：

```nginx
server {
    listen 80;
    server_name example.com;

    root /opt/no_starving/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 120s;
        proxy_send_timeout 120s;
        add_header X-Accel-Buffering no;
    }
}
```

启用配置：

```bash
sudo ln -s /etc/nginx/sites-available/no-starving.conf /etc/nginx/sites-enabled/no-starving.conf
sudo nginx -t
sudo systemctl reload nginx
```

SSE 流式接口依赖 Nginx 不缓冲响应，因此 `/api/` 反代必须保留 `proxy_buffering off`。

生产环境应再配置 HTTPS，例如通过云厂商证书或 Certbot。

### 10.1 后端健康检查

```bash
curl -i https://example.com/api/health
```

期望：

```text
HTTP/1.1 200 OK
```

响应体：

```json
{"code":200,"message":"success","data":{"status":"ok"}}
```

```bash
curl -N -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{"message":"我想吃鸡肉，推荐一道简单的菜"}' \
  https://example.com/api/chat/stream
```

期望可以看到类似事件：

```text

event: delta
data: {...}

event: done
data: {...}
```

### 10.3 前端页面验证

1. 打开 `https://example.com`。
2. 输入中文问题，例如“我想吃鸡肉，推荐一道简单的菜”。
3. 确认页面能流式展示回复。
4. 确认推荐页签能展示菜谱卡片。
5. 连续快速打开两个页面同时提问时，应有一个请求返回“当前已有请求处理中，请稍后再试”。

## 12. 运维注意事项

1. 本项目首版是单用户单 Agent 设计，不支持多用户并发会话。
2. `conversation_lock` 只在单进程内保证单活跃请求；如果用多进程 uvicorn 或多副本部署，需要改为数据库/Redis/分布式锁后再上线。
3. 本地 JSON 是首版真实持久化来源，必须备份 `backend/data`。
4. 如果 `agent-memory.json`、`taste-profile.json` 或 `recommendation-history.json` 损坏，后端会按空数据恢复，但用户画像和历史会丢失。
5. TheMealDB / TheCocktailDB 不是本系统可控服务，外部接口失败时前端应展示 Tool 失败或空结果提示。
6. OpenRouter 超时或模型不支持 Tool Calling 会影响主链路，需要通过日志和 `/api/health` 先确认基础配置。
7. 日志排障优先看 `requestId`、Tool 状态、OpenRouter 请求状态和 JSON 读写异常事件，不要直接打开完整用户消息日志。
8. 当前前端请求路径为同源 `/api/chat/stream`，不建议直接跨域访问后端。

## 13. 回滚方案

### 13.1 前端回滚

保留上一版 `frontend/dist` 产物目录或制品包。回滚时将 Nginx `root` 指向上一版构建产物，然后执行：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 13.2 后端回滚

后端回滚到上一版代码后重新安装依赖并重启服务：

```bash
cd /opt/no_starving/backend
.venv/bin/pip install -e .
sudo systemctl restart no-starving-backend
```

### 13.3 数据回滚

如需回滚画像或 memory，从备份恢复：

```text
backend/data/taste-profile.json
backend/data/recommendation-history.json
backend/data/agent-memory.json
```

恢复前应先停止后端服务，避免运行中写入覆盖恢复结果。

## 14. 常见问题

### 14.1 `/api/health` 返回 503

优先检查：

1. `OPENROUTER_API_KEY` 是否存在且非空。
2. `OPENROUTER_MODEL` 是否存在且非空。
3. systemd `EnvironmentFile` 路径是否正确。
4. 服务是否以预期工作目录 `/opt/no_starving/backend` 启动。

### 14.2 前端能打开但 Chat 失败

优先检查：

1. Nginx `/api/` 是否正确反代到 `127.0.0.1:8000`。
2. 后端服务是否运行。
3. 浏览器 Network 中 `/api/chat/stream` 是否返回 200 或明确 JSON 错误。
4. Nginx 是否关闭 `proxy_buffering`。

### 14.3 流式输出变成一次性返回

通常是代理层缓冲导致。检查 Nginx 配置：

```nginx
proxy_buffering off;
proxy_cache off;
add_header X-Accel-Buffering no;
```

### 14.4 两个页面不能同时聊天

这是首版设计限制。后端 `conversation_lock` 会保护单用户单 Agent，避免两个页面同时写同一份本地 JSON memory。

### 14.5 禁酒偏好没有跨设备同步

首版没有账号系统，画像仅保存在当前后端实例的本地 JSON 中，不做跨设备、跨实例同步。
